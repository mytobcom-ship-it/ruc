MapMatchSvr 알고리즘 플로우차트

MapMatchSvr/bin/config.ini 및 실제 소스를 근거로 재구성한 제어흐름입니다. 함수명 대신 우리말 설명으로 표현했고, 합칠 수 있는 단계는 하나로 묶었습니다. (2026-07-22 기준 소스)

MATCHED / 성공 SKIP / 저신뢰 ERROR / 완전실패 / 배제 중간 처리 단계

0. 서버 시작

flowchart TB
    Main(["서버 시작"]) --> Init["초기 설정 로딩<br/>(DB 연결·도로데이터·처리기 준비)"]
    Init --> InitOK{"모두 성공?"}
    InitOK -->|실패| Rollback(["종료"])
    InitOK -->|성공| Recover["미완료 건 복구<br/>(최대 3회 재시도)"]
    Recover --> RecoverOK{"복구 성공?"}
    RecoverOK -->|실패| Rollback
    RecoverOK -->|성공| Running(["정상 운영 시작<br/>(수집·처리 흐름은 §1)"])

1. 수집 → 배치처리

config: [feeder] limit/fetch_interval/queue_pause_count/queue_max_count/queue_busy_min/queue_busy_max, [threads] count
쉬운 이름: 큐 크기 따라 수집 + 배치로 묶어 일괄 처리·저장
flowchart TD
    Start(("주기적으로 확인")) --> Q["대기 건수 확인"]
    Q --> P1{"대기건수 400 미만?"}
    P1 -->|Yes| Fetch["새 GPS 가져와서<br/>담당 처리기 대기열에 배분"]
    P1 -->|"No, 혼잡"| Wait1["이번엔 건너뜀"]
    Fetch --> NextPoll
    Wait1 --> NextPoll
    NextPoll["대기건수 따라<br/>0.5초~10초 후 재확인"]
    NextPoll -.->|반복| Start
    Fetch -.->|쌓임| WStart

    subgraph W["처리기(워커) — 여러 명 동시에"]
        direction TB
        WStart["대기열에서 GPS 묶음 하나 꺼냄"] --> WLoop["묶음 안 GPS를 한 건씩<br/>순서대로 판정 (§2)"]
        WLoop --> WSave["결과를 한 번에 저장 시도"]
        WSave --> WOK{"저장 성공?"}
        WOK -->|성공| WCommit["진행상태 확정<br/>(다음 묶음이 이어받음)"]
        WOK -->|실패| WRevert["이번 묶음 통째로 무효화"]
        WCommit --> WStart
        WRevert --> WStart
    end

    classDef warn fill:#fb8c00,color:#fff,stroke:#e65100;
    class P1 warn;
같은 차량(device)의 GPS는 항상 같은 처리기로 감 — 진행상태를 한 곳에서만 관리하기 위함입니다. 묶음(batch) 저장은 전부 성공 또는 전부 무효화 — 건별로 나눠 저장하지 않습니다.

2. 맵매칭 판정 파이프라인

config: [mapmatch] radius_skip/speed_factor/speed_margin/maxstep/distance, [worker] ttl_sec
쉬운 이름: 사전검증 → 방식선택 → 탐색
flowchart TD
    In(["GPS 1건 입력"]) --> Gate{"사전검증 통과?<br/>(형식·좌표·오차·이상속도)"}
    Gate -->|불통과| SkipEarly(["SKIP: 판정 진입 안 함"])
    Gate -->|통과| ModeCheck{"직전에 이어갈<br/>링크가 있나?"}

    ModeCheck -->|있음| Chain["이어찾기:<br/>그 주변부터 탐색"]
    Chain --> ChainLoop{"이번 범위 후보가<br/>충분히 좋은가?"}
    ChainLoop -->|"아니오(범위 더 넓힐 수 있음)"| ChainExpand["탐색범위 한 단계 확장"]
    ChainExpand -.->|반복| Chain
    ChainLoop -->|"예 / 한계 도달"| ChainOK{"성공?"}

    ChainOK -->|성공| Success(["매칭 성공"])
    ChainOK -->|실패| Fresh["새로찾기:<br/>반경 안 전체에서 처음부터"]
    ModeCheck -->|없음| Fresh

    Fresh --> FreshOK{"성공?"}
    FreshOK -->|성공| Success
    FreshOK -->|실패| Wider{"반경을 더<br/>넓힐 수 있나?"}
    Wider -->|있음| WiderTry["넓혀찾기:<br/>반경만 키워 재시도"]
    WiderTry --> WiderOK{"성공?"}
    WiderOK -->|성공| Success
    WiderOK -->|실패| RefOnly
    Wider -->|없음| RefOnly
    RefOnly["참고만찾기:<br/>방향·반경 무시하고 근처 좌표만 확보"]
    RefOnly --> Fail(["매칭 실패<br/>(좌표는 참고용으로 남을 수 있음)"])

    classDef ok fill:#1e88e5,color:#fff,stroke:#0d47a1;
    classDef fail fill:#e53935,color:#fff,stroke:#b71c1c;
    class Success ok;
    class Fail fail;
"이어찾기"의 탐색범위 확장은 조기확정 최적화입니다 — 후보가 도로 끝점에 걸리거나 방향이 많이 어긋날 때만, 여유가 있으면 한 단계 더 넓혀 비교합니다(maxstep 설정이 상한). 성공 후에도 방향이 심하게 안 맞으면 "새로찾기"로 한 번 더 대조해 더 나은 갈림길 후보가 있는지 확인합니다.

3. 후보 탐색·채점·역행판정

config: [mapmatch] radius/radius_scale/min/max, alt_gap/penalty/weight/slope, reverse_weight/speed/margin
쉬운 이름: 후보 찾기 → 점수 매기기 → 역행 감점
"새로찾기"는 주변 여러 방향을 뒤져 후보를 모으고, 모인 후보 하나하나를 아래 기준으로 채점합니다. "이어찾기"도 매 단계마다 같은 채점 기준을 씁니다. 
flowchart TD
    Cand(["후보 지점 하나 검토"]) --> Dir{"진행방향이 도로와<br/>120도 넘게 다른가?"}
    Dir -->|예| Excl(["제외"])
    Dir -->|아니오| Proj["도로 위 가장 가까운<br/>지점까지 거리 계산"]
    Proj --> RadCheck{"그 거리가<br/>탐색반경보다 먼가?"}
    RadCheck -->|예| Excl
    RadCheck -->|아니오| Score["기본점수 = 거리 + 방향차이 + 고도차이"]

    Score --> Rev1{"같은 도로에서<br/>직전보다 뒤로 갔나?"}
    Rev1 -->|예| Pen1["되돌아간 거리만큼 감점<br/>(저속이면 약간 봐줌)"]
    Rev1 -->|아니오| Rev2{"직전 도로 끝점으로<br/>거꾸로 들어가는 도로인가?"}
    Rev2 -->|예| Pen1
    Rev2 -->|아니오| Collect
    Pen1 --> Collect["후보 목록에 추가"]
    Collect --> Pick["가장 점수 낮은 후보 채택"]

    classDef warn fill:#fb8c00,color:#fff,stroke:#e65100;
    class Pen1 warn;
역행 감점은 "GPS 노이즈로 매칭점이 앞뒤로 튀는 것"을 억제·구분하려는 목적이며, 일방통행 역주행 자체를 판정하는 기능은 아닙니다 — 도로 데이터에 일방통행 여부 필드가 없기 때문입니다. 고도차이 채점은 터널 구간이거나 GPS 고도가 비현실적일 땐 적용하지 않습니다.

4. 최종 판정

config: [mapmatch] timeout, reverse_confirm
쉬운 이름: MATCHED / SKIP / ERROR 결정
flowchart TD
    Start(["판정 시작"]) --> M1{"매칭 실패했지만<br/>근처 참고좌표는 있음?"}
    M1 -->|예| SKIP1(["SKIP: 반경밖 최근접"])
    M1 -->|아니오| M2{"매칭 완전 실패?"}
    M2 -->|예| ERROR1(["ERROR: 완전실패"])
    M2 -->|"아니오(매칭 성공)"| Check["아래 4가지를 확인<br/>(나중 해당 항목이 결과를 덮어씀)"]
    Check --> Decide{"해당하는 것 중<br/>가장 나중 항목은?"}
    Decide -->|"④ 역행 판단 애매"| SKIP4(["SKIP: 판단불가"])
    Decide -->|"③ 도로끝 저신뢰"| SKIP3(["SKIP: 저신뢰 매칭"])
    Decide -->|"② 연속역행 미확정"| SKIP2(["SKIP: 역행 미확정"])
    Decide -->|"① 처리시간 초과"| ERROR2(["ERROR: 시간초과"])
    Decide -->|해당없음| MATCHED1(["MATCHED: 정상 저장"])

    classDef ok fill:#1e88e5,color:#fff,stroke:#0d47a1;
    classDef warn fill:#fb8c00,color:#fff,stroke:#e65100;
    classDef fail fill:#e53935,color:#fff,stroke:#b71c1c;
    class MATCHED1 ok;
    class SKIP1,SKIP2,SKIP3,SKIP4 warn;
    class ERROR1,ERROR2 fail;
MATCHED로 확정될 때만 위치·시각·고도 기준점이 다음 GPS 처리에 쓰일 값으로 전진합니다. 다만 도로(링크) 자체를 따라가는 내부 추적은 SKIP이어도 계속 이어집니다 — 저장값(신뢰도 표시)과 내부 추적은 따로 관리됩니다.

5. 세션 피드백 루프 — "순차 진행이 아닌" 진짜 루프 구조

flowchart LR
    subgraph N["GPS(n) 처리"]
        direction TB
        P1["GPS 1건 판정 (§2~§4)"] --> P2["진행상태 조건부 갱신"]
    end
    subgraph N1["GPS(n+1) 처리"]
        direction TB
        Q1["방식선택·보정·역행판정·<br/>이상속도 검사의 기준점으로 사용"]
    end
    P2 -->|"진행상태 전달"| Q1
    Q1 -.-> Next(("GPS(n+2) …<br/>동일 구조로 반복"))

    classDef fb fill:#1e88e5,color:#fff,stroke:#0d47a1;
    classDef loop fill:none,stroke:#94a3b8,stroke-dasharray: 3 3,color:#64748b;
    class P2 fb;
    class Next loop;
진행상태이번 처리 갱신 조건다음 처리 사용처
마지막 매칭 도로매칭 성공 시 항상 전진 (SKIP이어도 전진)이어찾기 시도 여부 결정
도로 위 마지막 위치역행 의심 없거나 확정됐을 때만 전진같은 도로 역행 감점 기준점
마지막 매칭 좌표·시각신뢰할 수 있는 매칭일 때만다음 GPS 방향/속도 보정 기준점
직전 고도·도로유형위와 동일 + 고도값 유효할 때고도차이 채점 기준점
연속 역행 횟수역행 의심이면 +1, 아니면 0으로 리셋"역행 미확정 SKIP" 판정 기준
직전 처리 신뢰 여부신뢰할 수 있는 매칭일 때만 참이상속도 검사의 전제조건
trip 식별자매 처리마다 갱신trip 종료 누락 감지 기준
GPS 포인트는 독립적으로 처리되지 않습니다. 매칭 성공 여부·역행 여부·신뢰도에 따라 진행상태가 조건부로 전진하거나 고정되며, 이 상태가 다음 포인트의 판정을 전부 좌우하는 피드백 루프입니다.
